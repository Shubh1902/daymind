Plan Generation Flow

How DayMind builds your daily schedule from scratch using Claude with extended thinking.

Triggered on dashboard load · Cached for the day · Force-regeneratable
1

User visits /dashboard

Next.js server component calls generateDayPlan(userId)

File: app/(app)/dashboard/page.tsx

This is a Server Component — no client JS needed for this step. The function is imported from app/actions/ai.ts and called with "user_me". If a plan already exists in DailySession for today, it's returned immediately without calling Claude.
2

Cache check in Postgres

Query DailySession for today. If plan exists → return immediately (no Claude call).

// One DB query to check today's plan
const existing = await prisma.dailySession.findFirst({
  where: { userId, date: { gte: today, lt: tomorrow } }
})

if (existing?.plan) {
  return { id: existing.id, briefing: existing.briefing,
             plan: existing.plan as PlanItem[], questions: [] }
  // ✅ Claude NOT called — cached result returned
}
This cache means Claude is only called once per day per user, unless the user clicks "Regenerate".
3

Fetch tasks + user context

Parallel DB queries — incomplete tasks ordered by deadline, plus any stored context answers.

const [tasks, contexts] = await Promise.all([
  prisma.task.findMany({
    where: { userId, completed: false },
    orderBy: [{ deadline: "asc" }, { createdAt: "asc" }],
  }),
  prisma.userContext.findMany({ where: { userId } }),
])
UserContext stores things like "How long does writing a proposal take you?" — Claude's previous questions that you've answered. These shape the schedule.
4

Build the user prompt

Tasks serialised to readable text with IDs, deadlines, priorities, defer counts.

Each task becomes a line like:
- ID: clx1a | "Write Q1 report" | Due: Mar 25 | Priority: high | Deferred 1x | Est: 90min

Then wrapped with current date/time and user context, with an explicit JSON schema Claude must follow. 
// The exact JSON shape Claude must return:
{
  "briefing": "2-3 sentences about today",
  "plan": [{
    "taskId": "exact-id-from-above",
    "scheduledTime": "9:00 AM",
    "estimatedMinutes": 30,
    "reason": "one sentence why this slot"
  }],
  "questions": ["Context questions to ask the user"]
}
5

Call Claude with extended thinking

Direct Anthropic SDK with thinking: { type: "adaptive" } and streaming.

const stream = anthropic.messages.stream({
  model: "claude-sonnet-4-6",
  max_tokens: 4096,
  thinking: { type: "adaptive" },  // 🧠 Extended thinking on!
  system: SYSTEM_PROMPT,
  messages: [{ role: "user", content: userPrompt }],
})

const response = await stream.finalMessage()
💡 Why extended thinking?
With adaptive thinking, Claude decides whether to reason step-by-step before answering. For scheduling, this means it can consider trade-offs (deadline urgency vs. energy levels, dependency ordering) before locking in time slots. The thinking tokens don't appear in the output — only the final schedule JSON does.
6

Parse JSON + save to DB

Extract JSON from Claude's response, validate, save DailySession to Postgres.

// Find the JSON blob in the text response
const rawText = response.content
  .find(b => b.type === "text")?.text ?? ""
const jsonMatch = rawText.match(/\{[\s\S]*\}/)
const planData = JSON.parse(jsonMatch[0])

// Upsert the DailySession
const session = await prisma.dailySession.create({
  data: {
    userId,
    date: today,
    briefing: planData.briefing,
    plan: planData.plan as object[],
    userInputs: [],
  }
})
The plan is stored as Postgres JSON so future requests just hit the DB, not Claude.
7

Render on dashboard

DayPlan component visualises the schedule; ContextQuestions shows Claude's follow-up questions.

The returned DayPlanResult contains:
  • briefing — 2-3 sentence summary shown at top of dashboard
  • plan[] — array of scheduled items rendered as a timeline
  • questions[] — shown by ContextQuestions component for user to answer (saved back to UserContext table)

2

Serialises tasks and current plan into a rich system prompt for Claude.

// /api/chat/route.ts
const { messages, tasks, currentPlan, sessionId } =
  await request.json()

const tasksText = tasks
  .map(t => {

    if (t.priority !== "medium") line += ` | Priority: ${t.priority}`
    if (t.deferCount > 0) line += ` | Deferred ${t.deferCount}x`
    return line
  })
  .join("\n")

4

Hidden JSON block extracted

Claude always ends responses with a <plan_update> XML block. Frontend strips it from display.

What Claude sends (full text): 
// Visible conversational part:
Got it! I've pushed the investor report to Sunday and added
an email task for Sarah. Your afternoon is clearer now.

// Hidden machine-readable block:
<plan_update>
{
  "plan": [{"taskId":"clx2def","scheduledTime":"9:30 AM",...}],
  "briefing": "Updated briefing...",
  "newTasks": [{"text":"Email Sarah re: Q1 update","priority":"medium"}],
  "completedTaskIds": [],
  "deferredTaskIds": ["clx1abc"]
}
</plan_update>
🎭 The clever trick
stripPlanUpdate(text) removes the XML block before displaying the message bubble. The user sees only the natural conversation, not the JSON machinery.
5

onFinish: apply DB mutations

After streaming completes, the route applies task creates, completions, deferrals, and plan updates.

// Three possible mutations, run in parallel:

// 1. Create new tasks
if (update.newTasks?.length) {
  await Promise.all(
    update.newTasks.map(t => prisma.task.create({ data: {...} }))
  )
}

// 2. Mark tasks as completed
if (update.completedTaskIds?.length) {
  await prisma.task.updateMany({
    where: { id: { in: update.completedTaskIds } },
    data: { completed: true, completedAt: new Date() }
  })
}

// 3. Defer tasks (increment counter)
for (const id of update.deferredTaskIds) {
  await prisma.task.update({
    where: { id },
    data: { deferCount: { increment: 1 } }
  })
}
6

router.refresh() → UI updates

The onFinish callback in useChat triggers a Next.js router refresh, reloading server components.

router.refresh() causes Next.js to re-fetch all server components for the current route without a full page reload. Since DayPlan and ContextQuestions are server components, they re-render with fresh data from Postgres — reflecting whatever Claude just changed.

Prompts Deep Dive

The exact system prompts and how they shape Claude's behaviour in each context.

Plan Generation System Prompt

app/actions/ai.ts generateDayPlan() 
"You are a personal chief of staff. You know everything about the
user's tasks and context. Your job is to produce a realistic,
time-aware daily schedule. Be direct and practical. Never pad the
schedule — if something takes 20 minutes, say 20 minutes."
🎯 Why this prompt works

Three key constraints: role (chief of staff, not generic assistant), context omniscience (you know everything — no hedging), and anti-padding rule (prevents Claude from over-estimating to appear safe). The conciseness keeps the JSON tight.

Chat System Prompt

app/api/chat/route.ts POST /api/chat 
`You are DayMind, a personal chief of staff AI. You help the user manage their day.

CURRENT TASK LIST:
${tasksText}

CURRENT DAY PLAN:
${planText}

When the user sends a message:
- Respond conversationally in 1-3 sentences
- If they want to add a task, ignore a task, defer a task, or change
  priorities — acknowledge it and adjust the plan mentally
- Always end your response with a JSON block on its own line, like this:
<plan_update>
{"plan":[...],"briefing":"...","newTasks":[...],"completedTaskIds":[],"deferredTaskIds":[]}
</plan_update>

Rules for the JSON block:
- "plan": always return the FULL updated plan array (all tasks, reshuffled if needed)
- "briefing": short updated 1-2 sentence briefing reflecting changes
- "newTasks": array of tasks to create (only if user is adding something new)
- "completedTaskIds": task IDs to mark done
- "deferredTaskIds": task IDs to defer to tomorrow

Keep your conversational reply ABOVE the <plan_update> block. Do not explain the JSON.`
🎭 Dual-output pattern

This prompt instructs Claude to produce two simultaneous outputs in one response: a human-readable reply AND a machine-parseable JSON block. The XML wrapper (<plan_update>) makes regex extraction reliable. "Do not explain the JSON" prevents Claude from narrating the data.

Prompt Engineering Patterns Used

📐 Strict Output Schema

Both prompts provide an exact JSON shape Claude must return. This prevents hallucinated field names and makes parsing reliable.

🏷️ ID Anchoring

Tasks are sent with their real DB IDs (clx1abc). Claude must use these exact strings — no fuzzy matching needed when writing back to the DB.

🧱 Full Context on Every Request

The current task list AND plan are injected into every chat request, so Claude always has ground truth — no "Claude might have the wrong plan" bugs.

🚫 Negative Instructions

"Never pad the schedule" and "Do not explain the JSON" use negation to prevent common LLM tendencies. More effective than hoping Claude avoids them.

Data Shapes Claude Must Follow

// app/actions/ai.ts
export type PlanItem = {
  taskId: string            // must match a real Task.id in DB
  scheduledTime: string    // e.g. "9:00 AM"
  estimatedMinutes: number // integer minutes
  reason: string           // one sentence — shown in UI
}
export type DayPlanResult = {
  id: string               // DailySession.id
  briefing: string | null  // shown at top of dashboard
  plan: PlanItem[]          // ordered schedule
  questions: string[]      // Claude's follow-up questions

// What's inside <plan_update> in chat responses
{
  plan?: PlanItem[]          // FULL updated plan (always all items)
  briefing?: string         // updated 1-2 sentence briefing

    text: string

}

Streaming Status States

The useChat hook exposes a status field. Here's what each means in the UI:

idle

No request in flight. Input is enabled, send button is active.

submitted

Request sent, waiting for first token. Spinner shows, input disabled.

streaming

Tokens arriving. Text appears incrementally in the message bubble. Input still disabled. After the last token, onFinish fires → DB writes → router.refresh() → status returns to idle.
